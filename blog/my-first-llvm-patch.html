My First Patch to LLVM
Posted: 2026-04-16 | Author: Siddharth | Category: Open Source

A few weeks ago I got my first pull request merged into the LLVM project. The change is small. PR #181400 moves bf16div into a header-only implementation inside the shared math support layer. The diff is not impressive, but the process was.

Context

LLVM libc is gradually moving higher math functions toward header-only implementations. This is tied to C++23 constexpr math. If the implementation is available in headers, the compiler can evaluate it at compile time instead of emitting runtime calls.

There is an ongoing effort tracking this direction. bf16div, which implements division for bfloat16, was still defined in a .cpp file. Its logic directly invoked fputil::generic::div<bfloat16> and pulled in dependencies locally. This made it inconsistent with newer patterns.

The goal was to lift the implementation into src/__support/math/ and make the public entrypoint a thin wrapper.

What Changed

Before:

  • bf16div.cpp contained the full implementation
  • It directly called fputil::generic::div<bfloat16>(x, y)
  • Dependencies were listed manually in the entrypoint CMake target

After:

// libc/src/__support/math/bf16div.h
LIBC_INLINE bfloat16 bf16div(double x, double y) {
  return fputil::generic::div<bfloat16>(x, y);
}

// libc/src/math/generic/bf16div.cpp
LLVM_LIBC_FUNCTION(bfloat16, bf16div, (double x, double y)) {
  return math::bf16div(x, y);
}

The structure now matches other math functions: the actual computation lives in a header under __support; the exported symbol is a wrapper that forwards into the support layer; the entrypoint becomes trivial and easier to reason about.

Build system changes followed the same pattern: CMake now depends on libc.src.__support.math.bf16div instead of listing internal headers; Bazel adds a corresponding libc_support_library and libc_math_function. This reduces duplication and keeps dependency graphs aligned across build systems.

Review Notes

My first version tried to push constexpr into fputil::generic::div. That was wrong.

The reviewer pointed out that recent changes do not mark the generic template itself as constexpr. Doing so would affect a wide surface area and requires more careful auditing. The intended pattern is to move structure first, then enable constexpr in a controlled way.

I reverted that change and kept the wrapper simple.

Other fixes during review:

  • Removed redundant includes from the .cpp
  • Fixed alphabetical ordering in a CMake test file
  • Restored a test I accidentally deleted

Second round feedback was minor. Formatting in the Bazel file. After that, the patch was approved and merged.

Tools Used

Contributing to LLVM requires a robust toolchain. Here's what I used:

  • AddressSanitizer — Memory error detector for catching buffer overflows and use-after-free bugs
  • Valgrind — For deep memory debugging and leak detection when ASan wasn't enough
  • GoogleTest — LLVM's testing framework for writing unit tests
  • Bazel / Starlark — Google's build system. Starlark is the Python-like language for BUILD rules
  • Tulsi — Generates Xcode projects from Bazel build files
  • Lavender — Generate Visual Studio projects from Bazel — essential for navigating LLVM on Windows
